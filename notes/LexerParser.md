# Lexing vs. Parsing in Compilers

This document outlines the core concepts, differences, and historical context of lexing and parsing, with references to the **HaikuScript** compiler.

---

## Core Concepts

### Lexing (Lexical Analysis / Tokenizing)
* **Definition:** The process of converting a raw sequence of characters (source code text) into a flat array of meaningful units called **tokens**.
* **Input:** Raw source code string.
* **Output:** A flat list or stream of tokens.
* **Key Tasks:**
  * Groups characters into words/symbols (e.g., matching `"set"` or a number like `"10"`).
  * Classifies each token with a type (e.g., `ASSIGN`, `NUMBER`, `IDENTIFIER`).
  * Discards irrelevant characters like extra whitespace or comments.
  * Checks for basic vocabulary validation (e.g., invalid words).
* **HaikuScript Reference:**
  * Handled by the [`tokenize`](file:///c:/Code/Fizzbash/haikuscript/haiku-core.js#L102-L152) function in [`haiku-core.js`](file:///c:/Code/Fizzbash/haikuscript/haiku-core.js).
  * Example conversion:
    * *Source Code:* `"set x to ten"`
    * *Tokens Output:* `[ { type: "ASSIGN" }, { type: "IDENTIFIER", value: "x" }, { type: "NUMBER", value: 10 } ]`

### Parsing (Syntactic Analysis)
* **Definition:** The process of analyzing the flat stream of tokens to construct a hierarchical representation matching the grammatical rules of the language.
* **Input:** The flat array of tokens generated by the lexer.
* **Output:** An **Abstract Syntax Tree (AST)**.
* **Key Tasks:**
  * Analyzes how tokens relate structurally to one another.
  * Validates grammar correctness (e.g., ensuring a `loop` block is balanced with an `end`).
  * Creates a tree where parent nodes represent compound statements and children represent their arguments or body elements.
* **HaikuScript Reference:**
  * Handled by the [`parseProgram`](file:///c:/Code/Fizzbash/haikuscript/haiku-core.js#L155-L208) function in [`haiku-core.js`](file:///c:/Code/Fizzbash/haikuscript/haiku-core.js).
  * Example conversion:
    * *Tokens Input:* `[ { type: "ASSIGN" }, { type: "IDENTIFIER" }, { type: "NUMBER" } ]`
    * *AST Node Output:* `{ type: "AssignmentStatement", target: "x", value: 10 }`

## Comparison Summary
* **Lexer focus:** Words. It answers: *"Is this word/symbol valid and what category of speech does it belong to?"*
* **Parser focus:** Sentences. It answers: *"Are these words arranged in a valid sentence structure, and what does the sentence mean?"*
* **Lexing Analogy:** Identifying that `"dog"`, `"ran"`, `"the"`, and `"quickly"` are real words and labeling their parts of speech.
* **Parsing Analogy:** Rejecting `"ran dog the quickly"` because it violates the grammatical structure of the language, despite containing valid words.

## Historical & Modern Compiler Context

* **Historical Multi-Pass Compilers:**
  * Early systems had highly restricted RAM.
  * Lexing and parsing were separated into completely different physical execution passes.
  * Lexers would output tokens to tapes or drums, which the parser read in a separate pass.
  * Standard code generators like `lex` (1975) and `yacc` (1975) standardized this distinct pipeline.

* **Modern Compilers (e.g., Go, Rust, V8, LLVM):**
  * Still retain the logical boundary between a lexer and a parser.
  * **Hand-Written Parsers:** Most production compilers have transitioned away from generator tools to hand-written, recursive-descent parsers. This provides better error messages, syntax recovery (repairing errors to continue parsing), and performance.
  * **Incremental Parsing:** Tools like **Tree-sitter** (configured for syntax highlighting in your project under [`grammar.js`](file:///c:/Code/Fizzbash/haikuscript/grammar.js)) compile grammar to parse code dynamically as the user types, recalculating only the updated tree nodes.
  * **Lexerless/Scannerless Parsers:** Highly specialized parsers combine both phases into a single pass, but they are generally less common due to complexity and execution speed.
